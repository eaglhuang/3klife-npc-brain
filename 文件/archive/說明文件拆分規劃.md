## 1. 拆分原則

### 1.1 保留在 `README.md` 的內容

最終 `README.md` 只保留「第一次進來一定要知道」的資訊：

1. npc-brain 是什麼
2. 它和 pipeline / Cocos / LangGraph 的關係
3. 主要 API 入口
4. 一張總覽流程圖或架構圖
5. 最短 quick start
6. 文件導覽地圖

換句話說，`README.md` 不再承擔：

- 完整 Gemini / Local Llama fallback 細節
- 完整 Pinecone / Qdrant / upsert / query 教學
- 完整 LangGraph Studio / localtunnel / deployment 操作手冊
- 完整 data contract 與 Cocos 測試流程細節

### 1.2 新文件的拆分原則

每份新文件都要：

- 聚焦單一主題
- 只保留該主題最重要的資訊
- 避免和其他文件重複貼整段內容
- 優先用「連結」取代重複拷貝
- 檔名使用**繁體中文**
- 建立後必須加入 `server/server_docs_reference.md`
- 每份檔案都要有 `doc_id` HTML 註解

### 1.3 建議的篇幅控制

- `README.md`：**150 ~ 250 行**
- 子文件：**120 ~ 280 行**
- 如果某份子文件超過 ~300 行，應再考慮拆成兩份

## 6. `server_docs_reference.md` 維護規則

所有真正建立的新文件，都必須遵守：

1. 檔案第一行加上：

```md
<!-- doc_id: doc_server_service_XXXX -->
```

2. 建立後立刻更新：

- `<3KLife-root>/server/server_docs_reference.md`

3. `server_docs_reference.md` 只登記**已存在的實體文件**，不要先把未建立的規劃檔列進去。

4. 檔名可用繁體中文，但 `doc_id` 要維持機器可解析、連號且唯一。

5. 建議把這批拆分後的文件都歸在 `service` subtype，因為它們本質上是在說明 npc-brain service 本身。
