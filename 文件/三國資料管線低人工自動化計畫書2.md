## 二、自動化主幹線設計

新增 **一支** 主腳本（建議檔名 `run_data_pipeline_autopilot.py`，與既有 convergence loop 並存而非取代），它只做四件事：

1. 量測下游缺失。
2. 對缺失分桶（每桶都有對應的「補充腳本」或「校正腳本」）。
3. 套硬規則（alias 表 + 黑白名單 + trust-zone skip-index）決定每筆能不能自動派工。
4. 依優先序呼叫既有腳本，全部寫入 `canonicalWrites=false` 的 staged 區，等下一輪 round 收斂後才升級。

它**不**自己寫 canonical 真理，**不**自己改 alias / 黑白名單；所有狀態改變都必須走既有的 review / skill / human gate。

## 四、硬規則閘（alias × 黑白名單 × trust-zone）

把這三個現有檔案合成一個單一決策函式 `should_auto_dispatch(row) -> Decision`，三段式短路：

```
1. alias gate（最早，最便宜）
   - generalId 必須存在於 alias_dictionary 的 high/medium-confidence map
   - 若 row 的 aliasMatch 命中、但 reviewStatusByGeneral != accepted/auto-accepted → DROP
   - 沒過 alias gate 的所有後續工作一律封鎖（不只是 G1 桶）

2. blacklist gate（硬否決）
   - pair / claim / relationshipType / sourceRef 任一命中 trust-zone skip-index 的 blacklist
     → DROP，且寫一筆 negative-evidence 進 audit ledger
   - row 的 verdict 命中 policy.semantic.blacklistCandidateVerdicts → 直接歸 blacklist-candidate

3. whitelist / trust-zone gate（路由）
   - pair 在 stable / human-locked / skill-reviewed → 走 fast lane（只允許補 sourceRef、不改 type）
   - pair 在 review / accumulating → 走 propose-only lane（產 staged overlay, 等下一輪 skill review）
   - pair 完全未出現 → 走 cold lane（先送 G2 packet promotion 建立第一條 evidence，再進 review）
```

`apply_hard_rules` 全程 `canonicalWrites=false`，並輸出 `decision-ledger.jsonl`，每筆都帶 (`gateLayer`, `decision`, `reason`, `policyVersion`, `aliasMapVersion`, `skipIndexVersion`)。這是 handoff §「白黑名單與信任區」要求的可追溯硬約束。

## 五、自動找錯誤的判斷

讓 orchestrator 在 `measure_gaps` 之後跑四個低成本檢查，全部命中即上 `error-detected` bucket，停止 dispatch 直到 round 收斂或人工介入：

1. **alias drift**：alias map 對某 generalId 的 `reviewStatusByGeneral` 變化但 stable_bootstrap 對應欄位沒變 → flag `alias-status-drift`。
2. **blacklist 反覆**：同一 pair 在最近 N 輪 staged overlay 中出現超過 K 次，仍命中 blacklist → flag `blacklist-thrash`，自動降為 propose-only 並通知 trust-zone reviewer。
3. **canonical 偷渡**：staged overlay 寫出的檔案 sha 與 `*-canonical-*` 路徑下任何檔案 sha 重合 → flag `canonical-bleed`，立刻 abort（這是 handoff §「服務端不是證據來源」的紅線）。
4. **version 矛盾**：latest_data_refresh.py 報告 `dataVersion` 一致但 `artifactVersion` 不一致 → flag `artifact-basis-drift`，先跑 refresh 再 retry。

## 六、自動回饋上游主幹線

每一輪 round 結束（一次主幹跑完），orchestrator 必須寫三筆 round-level 證據，這也是 run_full_roster_convergence_loop.py 已有的 manifest／resume 機制可以直接 reuse：

1. `autopilot-round-<n>.summary.json`：每桶處理數、whitelist/blacklist 命中、staged overlay 路徑、staged proposal 數、預期 queue 收斂量。
2. `autopilot-round-<n>.feedback.jsonl`：把這輪 dispatch 後 queue 應減少而**沒減少**的 pair 抓出來，回饋給上游 (`source-event-packets / relationship-evidence / ready-events`) 作為下一輪 G2 cold-lane 的優先輸入。
3. `autopilot-round-<n>.review-handoff.md`：列出本輪自動化「不敢做」的 case（hard rule DROP / `error-detected`），含 evidence 路徑與建議的人工 review 動作；可由既有 review reviewer 直接吃。

這三個產物剛好對應 handoff 給下一位 AI 的工作指引（先看上游證據、再看 service、不要在 web 補）。

## 七、停止條件（避免 runaway）

主腳本必須讀下列 hard cap，全部命中之一就停：

- `--max-rounds N`（沿用 convergence loop 既有參數）
- 連續 2 輪 G2 candidate 數低於 `min-progress-threshold`（預設 50）
- 任一 round 出現 `error-detected` 桶
- `blacklist-thrash` 或 `canonical-bleed` 任一非空

停止後輸出 `autopilot-stop.evidence.json` 並 exit 非零，方便 CI / cron 偵測。
