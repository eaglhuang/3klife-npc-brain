# 武將基本資料從0到1的誕生

> 文件使用原則：
> - 本文假設你正在使用 standalone `3klife-npc-brain` repo。
> - `<repo-root>` 代表你自己 clone 下來的專案根目錄。
> - 若本文包含啟動或驗證指令，預設以 Docker 為正式開發環境來源；本機 Python / venv 僅用於 IDE debug、LangGraph dev 或臨時工具。

> 說明一個「可以被對話系統實際使用」的武將基本資料，如何從原始章回文字一路長成可供 npc-brain runtime 使用的 persona / keyword / context / readiness 資產。

## 先講最短版

一個武將從 0 到 1，不是把整本《三國演義》丟給模型，然後請它「自由發揮」。

真正的做法是把原始文本逐步壓縮成幾層穩定資產：

1. **先知道這個人是誰**
2. **再知道他做過哪些可信事件**
3. **再把事件轉成可用的 persona / keywords / contexts**
4. **最後驗證這些資料已足以支撐一輪對話 smoke test**

也就是說，所謂「武將基本資料」不是單一 JSON，而是一組能支撐 runtime 的最小知識包。

## 0 到 1 的真正原理

### 0：只有原始文本，還不能直接對話

在最初狀態，你可能只有：

- 章回段落
- roster
- 手工 seed
- 一些別名或字號表

這時候最大的問題是：

- 人物稱呼不穩定
- 事件邊界不清楚
- 關係與地點常混在敘事裡
- 模型若直接讀全文，很容易發明細節

所以「0」不是沒資料，而是**資料還不能直接拿來做人物行為或對話**。

### 0.2：先建立人物身份基座

這一層的工作是回答：

> 章回裡出現的這個名字、字、稱呼，到底是不是同一個人？

例如同一位人物可能同時以：

- 本名
- 字號
- 官銜
- 陣營內稱呼

出現。

如果這一步沒做好，後面事件、keyword、persona 都會綁錯人。

### 0.4：抽出「可信事件」

當人物身份能穩定對齊後，下一步不是直接產生台詞，而是先抽事件。

事件通常至少要回答：

- 誰做了什麼
- 在哪裡
- 和誰有關
- 依據來自哪幾段 source refs

但這些抽取不會全部都可靠，所以會分成：

- 可以直接往下走的 ready events
- 需要 review 的 candidates

### 0.6：把不穩定的候選經過 review / ABAB 精修

這一步的原理是：

- 先讓系統自動補能補的資訊
- 再讓人工或半人工 review 高風險項
- 把 accept / edit / reject / defer 的結果重新寫回

這樣做的好處是：

- 把模型自由發揮的空間壓小
- 讓人物資料不是「像」，而是「有來源依據」
- 為後面 persona 與 keyword 提供可信底座

### 0.8：把事件投影成 runtime 可用資產

光有事件還不夠，runtime 真正要吃的是：

- `keyword options`
- `persona card`
- `context options`
- `api readiness`

這代表系統不是直接把 event 原文拿去說話，而是把事件投影成：

- 這個人物的說話重點是什麼
- 他在什麼情境下比較會提哪些對象 / 地點 / 物件
- 他的語氣、立場、關係和禁忌大致是什麼

### 1：可以被對話系統使用

當一位武將到 1，不代表「資料完整到無可挑剔」，而是代表：

> 這份資料已足以支撐一輪真實的對話 smoke test，而且不需要 runtime 自己重新理解整本原文。

通常至少表示：

- 有穩定 `generalId`
- 有最小可用 persona
- 有可用 keyword options
- 有至少一批可用 context
- readiness 已通過基本檢查

## 一個最小可用武將基本資料，實際包含什麼？

可以把它想成這 5 類東西：

1. **身份層**
   - `generalId`
   - aliases / courtesy name / roster mapping

2. **事件層**
   - 經過抽取與 review 的可信事件
   - source refs
   - location / participants / relationship edges

3. **投影層**
   - keyword options
   - context options
   - persona card

4. **驗證層**
   - pilot refresh
   - API readiness
   - dialogue probe

5. **檢索層（可選但越來越重要）**
   - vector-ready records
   - 後續可進 Pinecone / Qdrant

## 簡單舉例：張飛如何從 0 長到可對話

這裡用一個簡化例子說明，不追求逐欄完整，只說原理。

### Step 1：原始段落

章回裡可能有一段大意像：

> 張飛於長坂橋前橫矛立馬，大喝一聲，曹軍不敢逼近。

這時候還只是文本片段。

### Step 2：身份對齊

系統要先確認：

- 這裡的「張飛」是不是 `zhang-fei`
- 是否和「翼德」等別名對得上

這一步完成後，人物身份才穩定。

### Step 3：抽事件

再把它抽成事件雛形，例如：

- eventKey：`changban-bridge-standoff`
- participants：`zhang-fei`, `cao-cao`（或曹軍）
- location：`長坂橋`
- source refs：對應章回段落

### Step 4：review / 精修

如果 location、關係、對象粒度不夠穩，就進 review：

- 是否要把對象寫成曹操本人，還是曹軍整體？
- 這是不是「威嚇阻敵」，還是「保護主公撤離」的事件？

review 完後，這筆事件才成為可信底層資產。

### Step 5：投影成 runtime 資料

從這筆事件往上長出：

- context option：`changban-bridge`
- keywords：`曹操`、`蛇矛`、`長坂橋`
- persona 傾向：勇烈、護主、臨戰強勢

這時候 runtime 已經可以知道：

- 張飛在什麼情境下常談什麼
- 哪些詞對他是有根據的
- 他的說話角度可能偏向什麼氣質

### Step 6：做 readiness 與 dialogue probe

最後跑 readiness，確認：

- context options 可讀
- keywords 可讀
- persona 可讀
- dialogue probe 不會完全失真

通過後，這位武將就從「只有原文存在」進化成「npc-brain 可以拿來對話」的狀態。

## 為什麼不能只存一份人物簡介就好？

因為對話不是只靠一句人設。

如果只有：

> 張飛，性格豪烈，擅長作戰。

這種摘要，會有幾個問題：

- 來源依據太薄
- 很難知道哪些詞能安全使用
- 很難知道他在不同情境下該怎麼說
- 很難和玩家行為、世界事件接起來

所以真正可用的「基本資料」，其實是一個**可驗證、可召回、可投影、可擴張**的結構，不是一段簡介而已。

## 這份 0→1 資料之後怎麼被拿去用？

當武將完成最小知識包後，後面就可以接：

- keyword options API
- context options API
- dialogue API
- vector export / semantic retrieval
- 後續的世界事件 / 玩家互動 decision engine

也就是說，這份資料不是為了「存檔好看」，而是為了成為：

> NPC 行為與對話決策的第一層底座。
